# project-management 开发规范

本文档描述本仓库的技术栈、目录约定、模块接入流程与编码规范，供新增功能（如「图片加解密」模块）时统一遵循。

---

## 1. 项目概览

| 层级 | 技术 | 说明 |
|------|------|------|
| 根目录 | Node.js ESM | `main.js` 启动，合包 node + web |
| 服务端 | Express 5 | `server/src/`，REST API 前缀 `/api` |
| 前端 | React 19 + TypeScript + Vite | `web/src/`，Less 模块化样式 |
| 构建 | `npm run build` | 前端产物输出到 `html/`，由 Express 静态托管 |
| 实时 | Socket.IO | 聊天、gomoku 等模块使用 |

**启动方式**

```bash
npm install          # 根目录，会联动 server / web
npm start            # 启动合包服务
npm run dev:web      # 仅前端热更新（需服务已运行或代理）
```

## 2. 目录结构

```
project-management/
├── main.js                 # 入口：设置根目录、加载 env、bootstrap
├── html/                   # 前端构建产物（勿手改）
├── server/src/
│   ├── app.js              # Express 实例
│   ├── bootstrap.js        # 静态资源、SPA fallback
│   ├── server/             # 各业务模块 API（按目录拆分）
│   │   ├── index.js        # 聚合 import 所有子模块
│   │   ├── upload/         # 文件上传
│   │   ├── file/           # 媒体列表、删除
│   │   └── ...
│   ├── middleware/         # 访问控制、鉴权
│   └── utils/              # 通用工具
└── web/src/
    ├── api/                # 前端 API 封装（post / upload）
    ├── components/         # 可复用组件
    ├── page/               # 页面（按模块分子目录）
    ├── Router/             # 路由配置与守卫
    ├── constants/          # 常量（模块权限等）
    ├── hooks/              # React Hooks
    ├── utils/              # 纯函数工具
    └── type/               # 共享类型
```

**约定**

- 新业务能力优先「前端页面 + 服务端 API」成对新增，避免把业务逻辑塞进 `main.js`。
- 可复用 UI 放 `components/`，仅单页使用的子组件可放在 `page/<Module>/` 下。
- 算法/纯逻辑优先放 `utils/`，前后端可各有一份；若需完全一致，后续可抽 `shared/`（当前仓库尚无，新增时需评估）。

## 3. 新增模块检查清单

以模块 key `imageCrypto`、路由 `/image-crypto` 为例，需同步修改以下位置：

### 3.1 前端路由

| 文件 | 操作 |
|------|------|
| `web/src/Router/config.ts` | 注册 path 与页面组件 |
| `web/src/utils/moduleMap.ts` | 增加 path → moduleKey 映射 |

### 3.2 首页入口

| 文件 | 操作 |
|------|------|
| `web/src/page/Home/index.tsx` | `entries` 数组增加卡片（moduleKey、name、desc、icon、path、accent） |

### 3.3 模块可见性与权限

| 文件 | 操作 |
|------|------|
| `server/src/middleware/access.js` | `LOCAL_MODULES` 增加 key |
| `web/src/utils/accessRules.ts` | `LOCAL_MODULES` 增加 key |
| `web/src/constants/moduleAccess.ts` | `CONFIGURABLE_MODULES` 增加项（若需在系统配置中隐藏/要求登录） |

按需决定：

- 是否加入 `PUBLIC_NEVER`（公网默认不可见）
- 是否要求超管才能写（参考 `image` / `television` 在公网 channel 的策略）

### 3.4 服务端 API

| 文件 | 操作 |
|------|------|
| `server/src/server/<module>/index.js` | 新建路由文件 |
| `server/src/server/index.js` | `import './<module>/index.js'` |
| `web/src/api/<module>.ts` | 封装 `post` / `upload` 调用 |

### 3.5 页面骨架

推荐结构：

```
web/src/page/ImageCrypto/
├── Layout/index.tsx          # 可选：多 Tab 时用 ToolPageLayout + ModuleNavLinks
├── Home/index.tsx            # 默认子页
├── Home/index.module.less
├── components/               # 模块内专用组件
└── utils/                    # 图像算法（Canvas / ImageData）
```

参考现有模块：

- 单页工具：`DataMock`、`PlaneEditor` → `ToolPageLayout`
- 多子路由：`Game/Layout` + `ModuleNavLinks`
- 媒体列表：`Image/Home` → `MediaGallery` + `uploadPic`

## 4. API 规范

### 4.1 路径与方法

- 统一前缀：`/api/<资源>/<动作>`
- 默认 **POST** + JSON body（与现有 `post()` 一致）
- 文件上传：`multipart/form-data`，使用 `upload()` / `baseUpload`

### 4.2 响应格式

```json
{
  "success": true,
  "code": 0,
  "msg": "",
  "data": {}
}
```

服务端使用 `ok(res, data, msg)` / `fail(res, httpStatus, code, msg)`（`server/src/utils/httpResponse.js`）。

前端使用 `web/src/api/client.ts` 的 `requestJson` / `uploadJson`，`code === 0` 或 `success === true` 视为成功；失败自动 `message.error`。

### 4.3 上传与存储

- 图片走现有 `POST /api/upload/uploadPic`，依赖系统配置中的 `picUploadPath`
- 注册信息由 `registerMedia` 写入媒体库
- **保存处理结果到服务器**：将 Canvas `toBlob` 或算法输出通过 `FormData` 调用 `uploadPic` 即可，无需单独新端点（除非需要元数据/会话持久化）

## 5. 前端编码规范

### 5.1 组件与样式

- 函数组件 + Hooks，默认 export 页面组件
- 样式：`*.module.less`，通过 `styles.xxx` 引用，避免全局污染
- 页面外壳：优先 `ToolPageLayout`；需要顶栏操作时传 `actions` / `headerActions`
- 子页切换：`ModuleNavLinks` + `useNavigate` / `useRouterIds`

### 5.2 状态与 API

- 列表加载：`useEffect` + 本地 `useState`，错误由 API 层统一 toast
- 复杂表单：按 `Config/Home`、`DataMock` 拆子组件
- 类型：共享类型放 `web/src/type/`，模块专用类型可放页面目录

### 5.3 图像处理（Canvas）

本仓库已有 Canvas 实践（`PlaneEditor/View`），图片加解密模块建议：

1. 用 **离屏 canvas** 或 `document.createElement('canvas')` 处理，避免污染预览 DOM
2. 大图注意 **分块处理** 或 **Web Worker**，防止主线程卡顿
3. 导出：`canvas.toBlob('image/jpeg', 0.95)` 对齐贴吧工具常见质量
4. 算法与 UI 分离：`utils/algorithms/*.ts` 纯函数，`components/*Panel.tsx` 只负责交互

### 5.4 路由

- 配置在 `web/src/Router/config.ts`，支持 `children` 与 `redirect`
- `RouteGuard` 根据 `moduleMap` 与 `visibleModules` 拦截无权限访问
- 写操作前用 `useAuth().canWriteModule(moduleKey)` 或 `ModuleAuthGate` 控制

## 6. 服务端编码规范

### 6.1 路由文件模板

```javascript
import app from '../../app.js';
import { ok, fail } from '../../utils/httpResponse.js';

app.post('/api/<module>/<action>', async (req, res) => {
  try {
    // ...
    ok(res, result);
  } catch (error) {
    fail(res, 500, 1, error.message || '操作失败');
  }
});
```

### 6.2 何时需要服务端算力

| 场景 | 建议 |
|------|------|
| 像素重排、幻影坦克显形、双图合成预览 | **前端 Canvas**（零上传隐私、与现有工具一致） |
| 保存结果到磁盘 | 复用 `uploadPic` |
| 隐写图「多轮预设搜索」会话历史 | 可选服务端：存 sessionId → 预设树 JSON（刷新可恢复） |
| 超大图批量处理 | 可选 Worker 线程或 `sharp`（需新增依赖） |

当前 `server/package.json` **无** 图像处理库；首版建议计算全在前端，服务端只做存储与会话（若需要）。

## 7. 权限与安全

- **模块级**：`LOCAL_MODULES` + `computeVisibleModules` / `computeModuleCapabilities`
- **配置级**：系统配置 → 模块访问（默认 / 要求登录 / 隐藏）
- **公网 channel**：`image` 等仅超管可写；新模块若涉及敏感内容，建议默认 `PUBLIC_NEVER` 或 `requireLogin`
- **免责声明**：图片加解密工具须在 UI 标明「非密码学安全、仅供学习/合法用途」，与社区工具惯例一致

## 8. 图片加解密模块 — 领域规范（草案）

### 8.1 功能分区

| Tab | 功能 | 核心输入 |
|-----|------|----------|
| 加解密 | 混淆 / 解混淆 | 图片、算法、密钥（可选） |
| 隐写图 | 合成 / 分离（幻影坦克） | 表图 + 里图，或单张隐写图 |
| 智能显形 | 用户选预设 → 迭代收窄参数 | 隐写图 + 多轮 UI 选择 |
| 双图合并 | 图层混合 + 参数 + 水印 | 图 A、图 B、曝光等 |

### 8.2 算法注册表（建议）

统一描述每种算法，供下拉框与校验使用：

```typescript
interface ImageAlgorithmDef {
  id: string;                    // 如 'xiaofanqie' | 'block' | 'row' | 'pixel' | 'mirage'
  label: string;
  direction: 'scramble' | 'steganography' | 'blend';
  needsKey: boolean;
  keyType?: 'number' | 'string';
  keyRange?: { min: number; max: number };
  keyHint?: string;                // 如 '0 ~ 1.618'
}
```

### 8.3 加解密算法（需密钥）

| ID | 名称 | 密钥 | 实现要点 |
|----|------|------|----------|
| `xiaofanqie` | 小番茄 | (0, 1.618) 数值 | Gilbert 曲线像素置换 |
| `block` | 块混淆 | 任意字符串 | 分块 + 伪随机打乱 |
| `row` | 行像素混淆 | 字符串 | 按行置换 |
| `pixel` | 逐像素混淆 | 字符串 | 全图像素索引打乱 |
| `picEncryptRow` | PicEncrypt 行模式 | (0, 1) 数值 | 兼容贴吧工具 |

加解密互逆：同一 `id` + `key` + `width/height` 可逆操作。

### 8.4 隐写图（幻影坦克 / 光棱坦克）

- **合成**：表图 + 里图 → 单张 PNG（灰度分离 + 像素交错）
- **分离 / 显形**：调色阶、对比度、背景色；参数空间连续，适合「预设网格 + 用户选择」而非暴力枚举
- **智能显形 UI**：
  - 每轮展示 N 组预设，每组 2 张候选（如浅色底 / 深色底显形）
  - 用户选中「更能区分两张图」的一组
  - 根据选择收窄参数区间，生成下一轮预设；历史轮次可回溯
  - 「保存」将当前候选图 `uploadPic` 到服务器

### 8.5 双图合并与水印

- 参数：曝光、对比度、饱和度、混合模式、不透明度等（映射到 Canvas `globalCompositeOperation` + 像素运算或 CSS filter 预览）
- 水印文本规则：用户为每项配置字母代号，如 `E+30 C-10`（Exposure +30, Contrast -10）
- 导出时绘制在图片角落或用户指定位置

### 8.6 数据流

```
用户选图 → FileReader / URL → ImageBitmap
    → algorithm(input, params) → ImageData
    → 预览 canvas
    → [保存] → toBlob → FormData → uploadPic → 媒体库
```

可选会话 API（智能显形）：

```
POST /api/imageCrypto/session/create   { imageId | base64 }
POST /api/imageCrypto/session/refine   { sessionId, selectedPresetIds }
POST /api/imageCrypto/session/history  { sessionId }
```

## 10. 测试与验收

- 手动：各算法用已知贴吧样例（固定密钥）往返加解密
- 压缩：JPEG quality 0.95 往返后对比（小番茄声称抗压缩）
- 大图：3000×3000 级别不冻结 UI（Worker 或进度条）
- 权限：local / lan / public 三 channel 下模块可见与写权限
- 智能显形：至少 3 轮收窄后能稳定选出可辨图
